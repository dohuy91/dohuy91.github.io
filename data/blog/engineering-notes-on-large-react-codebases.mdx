date: '2026-03-14'

summary: Practical rules I use to keep large frontend codebases understandable, changeable, and reviewable.
---

Most frontend codebases do not become hard because React is complicated. They become hard because responsibilities blur over time.

The first thing I look for is whether a file has a single reason to change. When a component owns fetching, formatting, feature flags, and layout, every edit becomes high risk. I prefer thin page-level orchestration and smaller components with obvious inputs.

The second rule is to make state transitions boring. If a reviewer cannot tell what changes after a user action without mentally simulating the whole tree, the boundary is wrong. Push transformations into small helpers and keep render code descriptive.

The final rule is to optimize for review, not just authorship. Code that is easy to write in one pass is often expensive for the next person to audit. Smaller units, stable naming, and explicit tradeoffs reduce that cost immediately.
